## 2. Phase 0：技术验证（2周）

### 2.1 目标

验证 Unity 2022.3 LTS → WebGL → 微信小游戏 的**包体**和**性能**是否满足要求。这是整个项目最大的技术风险（首包4MB限制），必须在全面开发前给出明确结论。

### 2.2 验证标准

| 验证项 | 通过标准（GO） | 不通过标准（NO-GO） | 验证方法 |
|--------|---------------|-------------------|---------|
| **首包大小** | WebGL构建产物 ≤ 3.5MB（留0.5MB给首屏资源） | > 4MB，且无法通过常规优化压到4MB以内 | 空项目+核心代码WebGL构建，测量产物大小 |
| **代码基线** | IL2CPP + Brotli压缩后代码 wasm ≤ 1.5MB | wasm > 2.5MB，剥离后仍超标 | 构建后分析 build.gz 大小 |
| **运行帧率** | 6个战斗单位（3v3）场景稳定 ≥ 25fps | < 15fps 或频繁卡顿 > 500ms | 微信开发者工具 + 真机（中低端Android）测试 |
| **内存占用** | 峰值内存 ≤ 200MB | > 300MB 或被微信强制回收 | 微信开发者工具 Performance 面板 |
| **加载速度** | 冷启动到首屏 ≤ 5秒（WiFi） | > 10秒 | 真机测试计时 |
| **微信API兼容** | 音频播放、本地存储、分包加载正常 | 核心API不可用或严重Bug | 微信开发者工具 + 真机验证 |
| **资源分包** | CDN分包资源可异步加载并正确显示 | 分包加载失败或超时 > 10秒 | 搭建简易CDN测试 |

### 2.3 每周交付物

#### 第1周（W1）：构建基线验证

| 日期 | 任务 | 交付物 | 负责人 |
|------|------|--------|--------|

### 2.4 GO/NO-GO 决策

#### ✅ GO：进入Phase 1

所有验证项均通过或接近通过（可通过后续优化解决）。

#### ⚠️ CONDITIONAL GO：带风险进入Phase 1

部分指标接近临界值，需在Phase 1中同步优化。明确列出需跟进的风险项和兜底方案。

#### ❌ NO-GO：调整技术方案

包体或性能严重不达标，需考虑：
1. **方案B**: 首包只含加载器（~200KB），游戏代码全部走分包加载
2. **方案C**: 放弃Unity，改用Cocos Creator / Laya 等更轻量引擎
3. **方案D**: 放弃微信小游戏，改为H5网页游戏

> **决策日**: 第2周五下午，CTO出具验证报告，CEO决策是否继续。

## 8. 风险应对

### Phase 0 风险

| 风险 | 应对 |
|------|------|
| 首包超4MB | 极端方案：首包只含加载器（~200KB），代码全走分包 |
| 真机性能不达标 | 降帧到20fps、限制同屏4单位、纯UI战斗 |
| Phase 0 NO-GO | 切换Cocos Creator重做（预估+4周），或放弃小游戏改H5 |

### Phase 1 风险

| 风险 | 应对 |
|------|------|
| 战斗系统改造延期 | 跳过功能延后，先保证基础战斗可运行 |
| UI开发阻塞 | 占位UI先行，复用Unity内置UI组件 |
| 工期紧张 | 遗物系统从Phase 1移到Phase 2 |

### Phase 2 风险

| 风险 | 应对 |
|------|------|
| 机制怪复杂度膨胀 | MVP先做3种，后续迭代加 |
| 数值不平衡 | 权重外置JSON，支持热更新 |

### Phase 3 风险

| 风险 | 应对 |
|------|------|
| 提审被拒 | 预留2天修复时间，提前准备隐私政策 |
| 真机兼容性问题 | 准备WebGL 1.0降级方案 |
| CDN加载失败 | 本地兜底资源（简化版） |
